# ADR-002: Hybrid Architecture — Binary + Shell Shim

**Date**: 2026-04-06  
**Status**: Accepted

---

## Context

A fundamental OS constraint: **a child process cannot modify its parent process's working directory**. When a terminal user types `pd myproject`, the intent is to change the current working directory of their running shell. The `cd` operation must be performed by the shell itself, not by an external process.

This means a standalone binary alone cannot implement the navigation feature. The original bash implementation worked because the `pd` command was a bash *function* sourced into the current shell — not an external process at all.

Three approaches were considered:

1. **Pure shell scripts per shell**: Maintain separate implementations in bash, nushell, and PowerShell. Full feature logic in each script.
2. **Pure binary, no navigation**: The binary only manages bookmarks; users pipe the output into `cd` manually (e.g., `cd (pd -x myproject)`). No convenience wrapper.
3. **Hybrid: binary + thin shell shim**: The binary handles all logic; a minimal shell-specific function handles the `cd` call.

## Decision

Use the **hybrid architecture** (option 3):

- The **Rust binary** implements all bookmark logic, path resolution, file I/O, and generates the shell init scripts
- A **thin shell-native function** (generated by `pd init <shell>`) is sourced into the user's shell; it intercepts navigation calls and performs the `cd`
- The shim is intentionally minimal — it contains no bookmark logic; it only distinguishes navigation from management calls and delegates accordingly

## Rationale

**Option 1 rejected** because:
- Three separate codebases with the same feature set is a maintenance burden
- Feature parity is hard to guarantee — bugs fixed in one shell must be manually applied to others
- Each shell's scripting language has different capabilities (e.g., nushell's `def --env`, bash's `export -f`)

**Option 2 rejected** because:
- It removes the primary UX benefit of the tool — `pd name` as a direct navigation command
- It requires users to know and type a different pattern in each shell

**Option 3 accepted** because:
- This is the same pattern used by widely-deployed tools: `zoxide`, `autojump`, `starship`, and others
- The shim is small enough (5–15 lines) that it poses negligible maintenance overhead
- The shim is *generated* by the binary (`pd init <shell>`), so adding a new shell requires only a new template in the binary — not a separate codebase
- All real logic remains in the single Rust codebase

## How the Shim Works (per shell)

**Nushell** (`def --env pd`):
- `def --env` allows the command to modify the caller's `$env.PWD`
- Navigation: calls `^pd get <name>`, captures path, calls `cd`
- Other ops: calls `^pd ...$args`

**Bash** (`function pd`):
- Navigation: calls `command pd get <name>` (bypasses function), captures path, calls `cd`
- Other ops: calls `command pd "$@"`

**PowerShell** (`function pd`):
- Navigation: calls `& pd.exe get <name>`, captures path, calls `Set-Location`
- Other ops: calls `& pd.exe @args`

## Consequences

- Users must run a one-time profile setup (source the init script) for each shell they use
- The shim must be kept up to date when the binary's interface changes — this is managed by re-running `pd init <shell>` on update
- Adding support for a new shell requires writing a new init template in the binary, not a new codebase
