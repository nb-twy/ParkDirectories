# ADR-001: Adopt Rust as the Core Implementation Language

**Date**: 2026-04-06  
**Status**: Accepted

---

## Context

The existing implementation of Park Directories is a bash function sourced into the user's shell. This works well in bash but is not portable to nushell or PowerShell, which have incompatible scripting languages. Maintaining three separate script implementations (bash, nushell, PowerShell) would triple the maintenance surface and make it difficult to guarantee feature parity.

A compiled binary avoids this: one codebase produces executables for Windows, Linux, and macOS from a single source tree.

## Decision

The core logic of Park Directories — bookmark management, file I/O, path resolution, shell init script generation — will be implemented in **Rust** and compiled to a native binary (`pd` / `pd.exe`).

Shell-specific code is limited to a thin integration shim (5–15 lines per shell), generated by the binary itself via `pd init <shell>`.

## Rationale

- **Cross-platform**: Rust's `cargo` toolchain compiles to Windows, Linux, and macOS targets from a single codebase
- **Single source of truth**: Features, bug fixes, and behavior are consistent across all shells
- **Static binaries**: Rust (especially with musl on Linux) produces self-contained executables with no runtime dependencies — ideal for dropping onto remote servers
- **Performance**: Startup time for a small Rust binary is negligible; no interpreter overhead
- **The developer is experienced with Rust**: No learning-curve cost
- **Testability**: The binary can be unit- and integration-tested independently of any shell

## Consequences

- A Rust toolchain (`rustup`, `cargo`) is required to build from source
- Pre-built binaries will be provided in GitHub Releases for the common targets, so end users do not need Rust
- A shell shim is still required in each target shell to perform the `cd` operation (see ADR-002)
- The shell-specific code footprint is intentionally minimal; all behavior logic belongs in the binary
